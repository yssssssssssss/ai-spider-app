# 持续观察固定页面 - 设计文档

**日期**: 2026-05-28  
**项目**: ai-taobao-app 持续竞品观察能力  
**阶段**: MVP 设计稿  

---

## 1. 项目背景与目标

现有系统已经支持一次性提交竞品采集需求，通过 AutoGLM 在手机端执行路径、截图、上传、LLM 设计/运营分析，并在后台任务结果页查看截图和分析内容。

新的需求是支持用户持续关注某个竞品页面或固定页面，每天采集一次，沉淀跨天对比，回答这类问题：

1. 最近一段时间这个页面持续在做什么？
2. 今天和昨天相比发生了什么变化？
3. 最近 7 天或 30 天内，页面的设计、运营策略、活动模块有什么发展趋势？
4. 是否有某些重点模块、促销机制、视觉表达持续出现或突然变化？

MVP 只解决“固定页面首屏的持续观察”，不扩展到关键词搜索、多页面链路或全页面滚动采集。

---

## 2. 已确认产品决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 观察对象 | 固定页面 | 例如淘宝百亿补贴、淘宝秒杀等稳定入口页 |
| 采集范围 | 页面加载完成后的首屏 | 不滚动，不采集长页面 |
| 执行频率 | 每天一次 | MVP 固定为日更 |
| 执行时间 | 每天 10:00 | 后续可开放配置 |
| 入口描述 | 自然语言进入路径 | 用户填写“打开淘宝，进入百亿补贴”等 |
| 调度方式 | 系统内置调度器 | FastAPI 启动后由 APScheduler 或等价轻量调度执行 |
| 失败重试 | 失败后 10 分钟重试一次 | 同一天最多 2 次 |
| 失败策略 | 当天两次失败后暂停观察计划 | 用户需要手动恢复 |
| 日常对比 | 与前一天有效截图对比 | 生成“今日变化” |
| 周期报告 | 与周期首日和周期内历史对比 | 生成 7 天、30 天报告 |
| 报告形式 | 人类可读报告 + 结构化 JSON | 文本用于阅读，JSON 用于图表和后续筛选 |
| 创建方式 | 手动创建观察计划 | MVP 不支持从已有任务一键转观察计划 |

## 3. MVP 范围

### 3.1 包含

1. 新增“持续观察”后台入口。
2. 支持手动创建固定页面观察计划。
3. 每天 10:00 为每个 active 计划创建一次观察运行。
4. 系统将用户填写的自然语言入口路径编译为标准 AutoGLM prompt。
5. AutoGLM 只采集目标页面首屏，得到一张有效主截图。
6. 截图复用现有图片入库、OSS URL、LLM 设计/运营分析链路。
7. 将有效截图关联为观察快照。
8. 生成当天摘要，并从第二天开始生成“对比昨天”的变化分析。
9. 自动刷新 7 天和 30 天周期报告。
10. 前端展示观察计划列表、计划详情、最新截图、每日摘要、周期报告和历史时间线。

### 3.2 不包含

1. 关键词搜索结果页观察。
2. 多页面、多路径、多竞品横向对比。
3. 页面滚动、长截图、全页面采集。
4. 像素级 diff 或热区标注。
5. PDF / Markdown 导出。
6. 从已有成功任务一键创建观察计划。
7. 多实例部署下的分布式调度锁。

## 4. 核心流程

### 4.1 创建观察计划

1. 用户进入后台“持续观察”。
2. 点击“新建观察计划”。
3. 填写：
   - 观察名称
   - 目标 App
   - 目标页面
   - 进入路径
   - 关注问题
4. 后端保存为 `watch_plans`，默认状态为 `active`。

### 4.2 每日自动采集

1. 调度器每天 10:00 扫描 `active` 观察计划。
2. 对每个计划按 `watch_plan_id + run_date` 幂等创建 `watch_runs`。
3. 后端生成一次内部采集任务 `tasks`，并启动 AutoGLM。
4. AutoGLM 执行固定页面首屏采集 prompt。
5. 图片入库后进入现有 OSS 上传、LLM 分析、近似页面跳过逻辑。
6. 从本次任务中选择有效目标页截图，写入 `watch_snapshots`。
7. 生成 `watch_daily_summaries`。
8. 刷新该计划的 7 天和 30 天 `watch_period_reports`。
9. 更新 `watch_runs.status = success` 和 `watch_plans.last_run_at`。

### 4.3 失败重试与暂停

1. 第一次运行失败时，`attempt_count = 1`，10 分钟后自动重试。
2. 第二次运行仍失败时，`watch_runs.status = failed`。
3. 同时将 `watch_plans.status = paused`。
4. `pause_reason` 记录失败原因，例如 AutoGLM 无法到达目标页、截图为空、LLM 分析失败等。
5. 用户在前端手动恢复后，计划重新进入下一次日程。

## 5. AutoGLM Prompt 模板

系统根据观察计划生成标准化 prompt：

```text
你正在执行固定页面观察任务。

目标 App：{target_app}
目标页面：{target_page}
进入路径：{entry_instruction}
关注问题：{focus_question}
采集范围：页面加载完成后的首屏。

执行规则：
1. 只采集目标页面首屏
2. 不要滚动
3. 不要进入商品详情页
4. 不要重复截图
5. 如果出现无关弹窗，关闭后继续进入目标页面
6. 如果弹窗本身与关注问题直接相关，先截图记录弹窗，再关闭弹窗继续进入目标页面
7. 完成目标页首屏截图后结束
```

设计原则：

1. 用户只填写自然语言入口路径，不直接编辑执行 prompt。
2. Prompt 中必须包含“只采集首屏”和“完成后结束”，减少过程页截图。
3. `focus_question` 会进入 prompt，影响 AutoGLM 对目标页和弹窗相关性的判断。

## 6. 数据库设计

### 6.1 `watch_plans` — 观察计划

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID, PK | 主键 |
| `name` | TEXT, NOT NULL | 观察名称 |
| `target_app` | TEXT, NOT NULL | 目标 App |
| `target_page` | TEXT, NOT NULL | 固定页面名称 |
| `entry_instruction` | TEXT, NOT NULL | 自然语言进入路径 |
| `focus_question` | TEXT | 用户关注问题 |
| `capture_scope` | TEXT, NOT NULL | MVP 固定 `first_screen` |
| `schedule_time` | TIME, NOT NULL | MVP 默认 `10:00` |
| `status` | TEXT, NOT NULL | `active` / `paused` |
| `pause_reason` | TEXT | 暂停原因 |
| `last_run_at` | TIMESTAMP | 最近一次完成时间 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 更新时间 |

### 6.2 `watch_runs` — 每日运行

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID, PK | 主键 |
| `watch_plan_id` | UUID, FK | 关联观察计划 |
| `task_id` | UUID, FK | 关联一次 AutoGLM 采集任务 |
| `run_date` | DATE, NOT NULL | 运行日期 |
| `attempt_count` | INTEGER, NOT NULL | 当天尝试次数 |
| `status` | TEXT, NOT NULL | `pending` / `running` / `success` / `failed` |
| `failure_reason` | TEXT | 失败原因 |
| `screenshot_count` | INTEGER | 原始截图数量 |
| `valid_snapshot_count` | INTEGER | 有效观察快照数量 |
| `created_at` | TIMESTAMP | 创建时间 |
| `completed_at` | TIMESTAMP | 完成时间 |

约束：

```text
UNIQUE(watch_plan_id, run_date)
```

这个约束是调度幂等的底线，防止服务重启或重复触发导致同一天重复采集。

### 6.3 `watch_snapshots` — 观察快照

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID, PK | 主键 |
| `watch_run_id` | UUID, FK | 关联每日运行 |
| `image_id` | UUID, FK | 关联图片主表 |
| `is_primary` | BOOLEAN | 是否为本次主截图 |
| `page_signature` | TEXT | 页面签名，用于近似页面判断 |
| `created_at` | TIMESTAMP | 创建时间 |

MVP 中每次成功运行应只有一张 `is_primary = true` 的有效目标页首屏截图。

### 6.4 `watch_daily_summaries` — 每日摘要

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID, PK | 主键 |
| `watch_run_id` | UUID, FK | 关联每日运行 |
| `summary` | TEXT | 综合摘要 |
| `design_summary` | TEXT | 设计变化摘要 |
| `ops_summary` | TEXT | 运营变化摘要 |
| `key_modules_json` | JSONB | 关键模块 |
| `promotions_json` | JSONB | 促销/利益点 |
| `changes_from_previous_json` | JSONB | 相比前一天变化 |
| `created_at` | TIMESTAMP | 创建时间 |

### 6.5 `watch_period_reports` — 周期报告

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID, PK | 主键 |
| `watch_plan_id` | UUID, FK | 关联观察计划 |
| `period_days` | INTEGER, NOT NULL | `7` / `30` |
| `date_from` | DATE, NOT NULL | 周期开始日期 |
| `date_to` | DATE, NOT NULL | 周期结束日期 |
| `report` | TEXT | 人类可读报告 |
| `structured_json` | JSONB | 趋势、变化点、持续动作等结构化内容 |
| `created_at` | TIMESTAMP | 创建时间 |

## 7. API 设计

### 7.1 观察计划

```text
GET    /api/admin/watch-plans
POST   /api/admin/watch-plans
GET    /api/admin/watch-plans/{id}
PATCH  /api/admin/watch-plans/{id}
POST   /api/admin/watch-plans/{id}/pause
POST   /api/admin/watch-plans/{id}/resume
POST   /api/admin/watch-plans/{id}/run-now
```

### 7.2 运行与结果

```text
GET    /api/admin/watch-plans/{id}/runs
GET    /api/admin/watch-runs/{id}
GET    /api/admin/watch-runs/{id}/snapshots
GET    /api/admin/watch-plans/{id}/reports?period_days=7
```

### 7.3 返回数据原则

1. 列表接口只返回摘要字段，避免把大段报告塞进表格。
2. 详情接口返回最新截图、今日摘要、昨日对比、7 天报告、30 天报告。
3. 所有截图继续复用 `ImageOut` 结构，保持 OSS URL、分析内容、跳过状态一致。

## 8. 分析流程

### 8.1 每日摘要 Prompt 输入

每日摘要基于：

1. 当天主截图的设计分析。
2. 当天主截图的运营分析。
3. 计划中的 `focus_question`。
4. 前一天有效快照和摘要，如果存在。

输出：

1. 今日页面重点。
2. 设计侧变化。
3. 运营侧变化。
4. 相比昨天新增、消失、强化、弱化的模块。
5. 结构化 JSON。

### 8.2 周期报告 Prompt 输入

周期报告基于：

1. 周期内每日摘要。
2. 周期首日与周期末日主截图分析。
3. 周期内所有结构化变化点。
4. 用户关注问题。

输出：

1. 最近一段时间持续在做什么。
2. 中间发生了哪些关键变化。
3. 哪些策略是长期稳定存在的。
4. 哪些模块或利益点是短期活动。
5. 对设计和运营的分别总结。

### 8.3 去重规则

现有一次性任务中已经有“近似页面只分析一张”的规则。持续观察中继续使用这个原则：

1. 同一次运行内，只保留目标页面首屏中的主截图。
2. 过程页、重复页、加载中页面、无关弹窗截图不进入观察快照。
3. 只有与关注问题直接相关的弹窗，才允许作为补充截图进入本次运行，但主截图仍必须是目标页面首屏。

## 10. 调度设计

MVP 使用系统内置调度器：

1. FastAPI 启动时注册每日 10:00 作业。
2. 作业只负责创建 `watch_runs` 和触发后台执行。
3. 创建运行时必须依赖数据库唯一约束保证幂等。
4. 重试任务可以由调度器的单次延迟任务或数据库轮询实现。
5. 服务重启时扫描 `pending` / `running` 超时任务并恢复为可重试或失败状态。

不做分布式锁的原因：

1. 当前项目是单机/本地运行形态。
2. 数据库唯一约束已经能挡住最关键的重复运行。
3. 过早引入 Redis 锁或分布式调度会增加维护成本。

## 12. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AutoGLM 到达目标页不稳定 | 每日观察失败或截图错误 | Prompt 固定化，失败重试一次，失败后暂停等待人工恢复 |
| 过程页截图混入分析 | 周期报告被噪声污染 | 继续使用近似页面跳过和目标页主截图筛选 |
| 调度重复触发 | 同一天产生多条运行 | `UNIQUE(watch_plan_id, run_date)` 做数据库幂等 |
| 页面样式小变化过多 | 报告抓不住真正变化 | 日报聚焦模块级变化，避免像素级 diff |
| LLM 成本增长 | 长期运行成本不可控 | 每日只分析一张主截图，周期报告基于摘要而不是反复分析全部图片 |
| 单机调度进程退出 | 当天任务漏跑 | 启动时扫描当天缺失运行并补偿创建 |

## 13. 验收标准

1. 可以手动创建一个固定页面观察计划。
2. 系统每天 10:00 自动创建一次观察运行，且同一天不会重复创建。
3. 立即运行可以生成 AutoGLM 任务并完成目标页首屏截图。
4. 有效截图能上传 OSS，前端能打开正确图片 URL。
5. 有效截图能完成设计分析和运营分析。
6. 每日摘要会结合关注问题生成并入库。
7. 第二次及之后运行能生成与前一天的变化分析。
8. 7 天和 30 天周期报告可以在观察详情页查看。
9. 当天连续两次失败后，观察计划自动暂停，并在前端展示暂停原因。

## 14. 关键决策记录

| 决策项 | 选择 | 理由 |
|--------|------|------|
| MVP 观察对象 | 固定页面首屏 | 用户已确认，边界清晰，成本最低 |
| 调度方案 | 系统内置调度器 | 当前项目不需要外部 cron 或分布式调度 |
| 执行时间 | 每天 10:00 | 用户已确认 |
| 失败策略 | 重试一次后暂停 | 避免连续失败消耗资源，也让问题显性化 |
| 报告存储 | 文本 + JSONB | 同时服务阅读和后续结构化展示 |
| 页面去重 | 只保留目标页主截图 | 解决 AutoGLM 过程页和重复截图噪声 |
